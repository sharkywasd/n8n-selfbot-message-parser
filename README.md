# n8n-selftbot
Telegram userbot for analyzing user messages with LLM. Supports local PostgreSQL DB & automatic result messages sending to Telegram chat

### Структура проекта:
`./`:
- `docs` - скриншоты для README.md и промпты `prompts` для генерации новых данных в проекте
- `n8n-output` - промапленная директория в docker-compose для передачи файлов из системы в контейнер (загрузка тестовых данных через файлы). Внутри директории находится ещё одна директория `test_data`, содержащая текущие тестовые сообщения в шаблоне Google Sheets
- `src` - основные файлы проекта
- `README.md`

`./src`:
- `workflows` - импорты воркфлоу проекта в .json формате
- docker-compose.yml
- env_example.txt - пример .env файла
- Palatine_selfbot - шаблон гугл таблиц


### Порты для контейнера
- 5678 (по умолчанию) - n8n
- 5432 (по умолчанию) - PostgreSQL

**Повторяйте нижеследующие щаги для последовательной установки и развёртывания проекта:**

**0. Регистрируем API Development Tools**

https://my.telegram.org/auth

1) Проходим авторизацию, вводим код. Попадаем на следующую страницу:

![Экран после авторизации](./docs/screenshots/tg_auth.png)

2) Нажимаем `API development tools`, попадаем в `Create new application`. Создаём новое приложение для получения API_ID и API_HASH
Заполняем все поля, вводим вразумительные названия, к примеру: 
- App Title: `Client Help Desk`
- App Shortname: `helpdesk1`
- URL: например, случайный сдаваемый в аренду (не занятый) домен, соответствующий региону регистрации аккаунта, я использовал `sviter.ru`
- Platform: Практичестки не имеет знаечния, я оставлял стандартный `Android`, можно попробовать `Web` или `Desktop`.
- Description: Имеющее смысл краткое описание на 10-15 слов. `Application for answering on client questions with of the automatic help desk`

3) Нажимаем `Create application`. На момент релиза рабочей версии (19.01.2026) Telegram до сих пор ведёт жестокий контроль за созданием данных приложений (получаем простое `ERROR` в ответ на нажатие кнопки), в  РФ осложнена регистрация новых аккаунтов. У меня получилось зарегистрировать приложение после насилия на кнопкой путём многократного нажатия кнопки и сброса ошибки с помощью клавиши `Enter` в течение 10 минут. Добро пожаловать в мир магии при регистрации приложения Telegram для получения api_id, api_hash, удачи!

**2. Запускаем контейнер**
В .yml файле описаны службы и образы, которые будут созданы и развёрнуты, в их список входят:
- заглушка для телеметрии Telepilot, которая мешает работе ноды (потому что баг Telepilot)
- БД postgresql
- n8n с предустановленной community-нодой Telepilot (@telepilotco/n8n-nodes-telepilot). В образе промаплена дополнительная папка `- ./n8n-output:/home/node/.n8n-files` в директории проекта, для чтения/записи файлов через n8n. При переименовании папки заменяем `n8n-output` в данной строке на название папки в директории проекта, которое вы ей задали.

Необходим установленный Docker и плагин Docker Compose:
https://docs.docker.com/compose/install/


1) **Настраиваем .env файл** (`./src/env_expample.txt`):
```bash
cd ./n8n-selfbot-message-parser/src
cp env_example.txt .env
nano .env
```
Достаточно изменить только пароль в Postgre `POSTGRES_PASSWORD`, но желательно также изменить `POSTGRES_USER`

- Параметры `N8N_LOG_LEVEL` и `N8N_LOG_OUTPUT` оставляем по умолчанию `info` и `console` (при необходимости изменяем: https://docs.n8n.io/hosting/logging-monitoring/logging/)


2) После настройки `.env` поднимаем контейнер:

```bash
docker compose -p "project-name" up -d
```
где:
- `-p` - флаг изменения названия проекта в докере (без фалага докер будет брать название папки, в которой лежит `docker-compose.yml`)
- `project-name` - задаваемое название проекта, которе будет префиксом для всех контейнеров, сетей и томов
- `up` - команда для запуска контейнеров
- `-d` - флаг, который позволяет открепить контейнер от терминала (то есть в терминале не будет видно логов, в терминале можно будет продолжать вводить новые команды, а если терминал закрыть, то контейнер продолжит работу)

**4. Работаем с n8n**

4.1. Поочерёдно создаём новые workflow и импортируем все остальные workflow с помощью .json файлов.
<br>4.1.1. `LLM Batch2` и `LLM Batch2` - обработка сообщения с помощью LLM
<br>4.1.2. `Regexp1`, `Regexp2`, `Regexp3` - обработка сообщения регулряным вырадением
<br>4.1.3 `chat_id Parser ID1`, `chat_id Parser ID2`, `chat_id Parser ID3` - воркфлоу для записи chat_id в лист Search'а для фильтрации чатов в сёрчах
<br>4.1.4 `Error Сatcher` и `DB Clean-up` - отправка сообщений об ошибке в чат TG и очиска нерелевантных сообщений каждые 30 минут
<br>4.1.5 `UserInfo Sender` - отправка релевантных сообщений в чат TG
<br>4.1.6 `Chat_Opener_3_minutes` - открытие чатов для получения обновлений

Если в новых workflow появляются со значком предупреждения (красный треугольник), то надо выбрать добавленные credentials, а если их нет, то придётся заполнить их руками по новой, обновить страницу и они должны подтянуться в остальные ноды в воркфлоу.

4.3 Пример описания столбцов для одного search-листа (`Search Speech`):
- `A (ID)` - ID строки/записи.
- `B (Speech (ID 1))` - данные по сёрчу (название, ключевые слова, промпт, тестовые записи).
- `I (tg каналы в которых искать)` - список каналов/чатов для поиска (`@username`), заполняется каждый с новой строки без лишних символов.
- `J (tg каналы chat_id)` - список `chat_id` для каналов из столбца `I`.
- `K (куда отправляем релевантные сообщения)` - целевой `chat_id` для отправки релевантных сообщений.
- `N (Техническая инфа)` - служебный столбец (заполняется автоматически).

## Обновление образа n8n
https://docs.n8n.io/hosting/installation/docker/#updating

С помощью Docker Desktop: нажимаем `Pull` на образе n8n, останавливаем и поднимаем контейнер. Старый образ можно удалить после успешного обновления.

С помощью docker compose:
1) Переходим в директорию проекта
2) `docker compose pull`
3) `docker compose down`
4) `docker compose up -d`

## Особенности:
- Избегайте слишком часты запросов к Telegram для избежания блокировки аккаунта.
- Может возникнуть проблема с работоспособностью ноды Telepilot при полном отключении сессии и очищении в чате с помощью команд `/stop`, `/clear`: `Chat not found` для чатов, которые на самом деле существуют. Для исправления проблемы вручную запускаем воркфлоу `chat_id Parser ID1` или испольуем ноды из Telepilot `Search public chat by username/title` (в случае конкретного группы/канала) или ноду-триггер `On new message` и отправляем на свой аккаунт сообщение с необходимого аккаунта (в случае личного чата). После полноценного выполнения воркфлоу ошибка должна исчезунть.
